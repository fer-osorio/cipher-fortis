# Test Infrastructure

The test suite for CipherFortis is organized in three layers of
responsibility. Each layer has a single job; nothing bleeds across
boundaries.

---

## Layer 1 — Raw I/O utilities (`TestUtils::IO`)

**Files:** `include/file_write_utils.hpp`, `src/file_write_utils.cpp`

Thin wrappers over `std::ofstream` and `std::ifstream`. These are the
*only* files in `tests/src/` that are allowed to open file streams
directly.

```cpp
namespace TestUtils::IO {
    // Write `size` pseudo-random bytes to `path`.
    void write_binary_file(const fs::path& path, size_t size);

    // Write `size` bytes generated by `gen(i)` to `path`.
    void write_binary_file(
        const fs::path& path,
        std::function<uint8_t(size_t)> gen,
        size_t size
    );

    // Write UTF-8 text to `path`.
    void write_text_file(const fs::path& path, const std::string& content);

    // Read the contents of `path` (binary or text mode).
    std::vector<uint8_t> read_file(
        const fs::path& path, bool binary = true
    );
}
```

Raster-image creation lives in `TestUtils::Raster`
(`include/raster_asset_utils.hpp`, `src/raster_asset_utils.cpp`) and is
the *only* place that calls `stbi_write_*`.

## Layer 2 — Asset factories (`AssetFactory` hierarchy)

**Files:** `include/asset_factory.hpp`, `include/*_asset_factory.hpp`,
`src/*_asset_factory.cpp`

Each concrete factory knows one file format and implements three
virtual methods:

| Method | Returns |
|--------|---------|
| `make_valid(dir)` | Path to a small, well-formed asset |
| `make_large(dir)` | Path to a large asset (performance tests) |
| `extension()` | Canonical extension string (e.g. `"bmp"`) |
| `is_binary()` | `true` for binary formats, `false` for text |

`make_corrupt(dir)` and `make_empty(dir)` are non-virtual helpers
provided by `AssetFactory` itself.

### Adding a new format

1. Create `include/<format>_asset_factory.hpp` and
   `src/<format>_asset_factory.cpp`.
2. Inherit from `AssetFactory` and implement the four virtual methods.
3. Add the `.cpp` to the `ciphfortis_test_fixtures` sources list in
   `CMakeLists.txt`.
4. Pass an instance of your factory to `SystemTests` (or any fixture
   that accepts a factory reference).

## Layer 3 — Environment lifecycle (`TestEnvironment`)

**Files:** `include/test_environment.hpp`, `src/test_environment.cpp`

RAII wrapper around a temporary directory. The constructor calls
`fs::create_directories`; the destructor calls `fs::remove_all`. All
fixtures hold a `TestEnvironment` as their **first** declared member
so that it is constructed before any path aliases and destroyed after
all other members.

```cpp
class TestEnvironment {
public:
    explicit TestEnvironment(const fs::path& base_dir);
    ~TestEnvironment();           // calls fs::remove_all(base_dir_)
    const fs::path& path() const;
    // non-copyable, non-movable
};
```

## Fixtures

| Class | Header | Data dir |
|-------|--------|----------|
| `FileBaseFixture` | `include/file_base_fixture.hpp` | `tests/data/file_base/` |
| `RasterImageFixture` | `include/raster_image_fixture.hpp` | `tests/data/raster_image/` |

Both fixtures expose a `testDataDir` alias (`const fs::path& testDataDir
= env_.path()`) for backwards-compatible access by test files.

---

## System tests (`SystemTests`)

**Files:** `include/system_workflows.hpp`, `src/system_workflows.cpp`,
`system/test_image_encryptorcpp.cpp`

`SystemTests` accepts an `AssetFactory` reference at construction time
and exercises the CLI binary end-to-end. All temporary files live under
`TestEnvironment env_("test_data")` which is cleaned up automatically
when the `SystemTests` object goes out of scope.

---

## Conventions

- `std::ofstream` / `std::ifstream` — only in `file_write_utils.cpp`.
- `stbi_write_*` — only in `raster_asset_utils.cpp`.
- `fs::create_directories` / `fs::remove_all` — only in
  `test_environment.cpp`.
- `.gitkeep` files — none; `TestEnvironment` owns the directory lifecycle.
